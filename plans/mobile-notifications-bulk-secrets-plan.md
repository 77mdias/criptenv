# Plan — Mobile Notification Dropdown Fix and Admin Bulk Secret Selection

**Date:** 2026-05-30  
**Status:** Proposed  
**Scope:** `apps/web` dashboard UX only; backend RBAC already guards human-session vault writes for `admin`/`owner`.

## Context

The current dashboard has two related UX gaps:

1. **Notification dropdown on mobile is broken**: `NotificationBell` renders an absolutely positioned panel with a desktop-first width (`w-[380px]`) anchored to the bell. Even with `max-w-[calc(100vw-1rem)]`, the panel can still feel clipped/misaligned in narrow dashboard layouts, especially when the top navigation has limited horizontal room.
2. **Secret deletion is one-by-one only**: admins/owners can delete a single secret via each row, but there is no bulk selection workflow for selecting individual secrets or all secrets in the active environment before deletion or future batch actions.

The current code already exposes the role needed for UI gates via `project.current_user_role`, and the secrets page computes `canManageSecrets` with `canWriteProjectSecrets()`. That same gate should protect the new bulk-selection UI.

## Goals

- Make the notification menu usable on mobile without horizontal overflow or clipping.
- Preserve the existing desktop dropdown visual style and behavior.
- Add admin/owner-only secret selection controls:
  - select one secret at a time;
  - select all secrets in the active environment;
  - clear selection;
  - delete selected secrets with a single confirmation dialog.
- Keep viewer/developer experience unchanged: no bulk checkboxes, select-all controls, or bulk destructive actions.
- Preserve zero-knowledge behavior: deletion still updates the encrypted vault client-side and pushes only encrypted vault data to the API.

## Non-Goals

- No backend schema changes.
- No new plaintext secret API endpoint.
- No cross-environment bulk selection in the first pass.
- No member/role management changes beyond reusing the existing `admin`/`owner` permission gate.
- No redesign of the full dashboard header.

## Implementation Plan

### 1. Notification dropdown mobile fix

**Files to inspect/update**

- `apps/web/src/components/layout/notification-bell.tsx`
- `apps/web/src/components/layout/__tests__/notification-bell.test.tsx`
- Potentially `apps/web/src/components/layout/top-nav.tsx` if the header container contributes to clipping.

**Approach**

- Keep the desktop behavior as an anchored dropdown at `sm`/`md` and above.
- Render the mobile panel as a viewport-safe fixed sheet/popover:
  - `fixed inset-x-3 top-16` or equivalent below the top nav;
  - `max-h-[min(70vh,480px)]` for the list area;
  - `w-auto` instead of a hard desktop width on small screens;
  - safe padding for notches if needed (`env(safe-area-inset-*)`) only if the project already uses that pattern.
- Ensure the panel remains above dashboard content with the existing `z-50` or a slightly higher z-index if needed.
- Keep `role="dialog"`, `aria-label="Notificações"`, outside-click close, mark-all, empty state, unread indicator, and click-through behavior.
- Consider body scroll only if the panel still causes awkward page scrolling; avoid adding global scroll locks unless necessary.

**Acceptance Criteria**

- On widths around 320px, 375px, and 430px, the notification panel is fully visible and horizontally contained.
- On desktop, the panel remains anchored to the bell with the existing width and visual hierarchy.
- The unread badge, loading state, empty state, and notification rows are unchanged functionally.
- Existing notification tests pass, with at least one new assertion covering the mobile-safe fixed/inset classes or behavior.

### 2. Bulk selection state for secrets

**Files to inspect/update**

- `apps/web/src/app/(dashboard)/projects/[id]/secrets/secrets-client.tsx`
- `apps/web/src/app/(dashboard)/projects/[id]/secrets/_components/unlocked-vault-panel.tsx`
- `apps/web/src/components/shared/secrets-table.tsx`
- `apps/web/src/components/shared/secret-row.tsx`
- `apps/web/src/app/(dashboard)/projects/[id]/secrets/use-project-secrets.ts`
- Existing focused tests under `apps/web/src/components/shared/__tests__/` and/or feature-local tests.

**State model**

- Store selected secret keys in the secrets client or a small feature-local hook:
  - `selectedSecretKeys: Set<string>` internally, exposed as `string[]` or callbacks to children;
  - reset selection when `activeEnv.id` changes;
  - prune selection when the decrypted secret list changes after import/delete/refresh;
  - clear selection when the vault is locked.
- Keep the source of truth key-based, not object-reference-based, because decrypted secret objects may be regenerated by vault refreshes.

**UI behavior**

- When `canManageSecrets` is true and the vault is unlocked:
  - show a checkbox per row;
  - show a select-all checkbox/control in the table/bulk toolbar;
  - show selected count;
  - show “Excluir selecionadas” destructive action only when `selectedSecretKeys.size > 0`;
  - provide “Limpar seleção”.
- When `canManageSecrets` is false:
  - do not render row checkboxes;
  - do not render select-all or bulk toolbar;
  - keep copy/reveal read-only actions available as today.
- On mobile:
  - row checkbox should appear before the secret key within the stacked row layout;
  - bulk toolbar should wrap cleanly, with full-width destructive button if needed.

**Accessibility**

- Use semantic checkboxes with labels such as `Selecionar secret DATABASE_URL`.
- Select-all checkbox label should describe scope: `Selecionar todas as secrets deste ambiente`.
- Indeterminate state should be visible when some but not all secrets are selected.
- Destructive bulk action should open the app `ConfirmActionDialog`, not `window.confirm`.

### 3. Bulk delete action

**Preferred implementation**

- Add a bulk delete action in `useProjectSecrets` that accepts keys or secrets:
  - `deleteSecrets(keys: string[])` or `deleteSelectedSecrets(selected: DecryptedSecret[])`;
  - validate `activeEnv`, `keyMaterial`, `vaultProof`, and loaded `secrets` exactly like single delete;
  - compute the next vault by filtering out selected keys once;
  - call `pushSecrets(nextSecrets)` once, not once per deleted secret.
- Keep single delete unchanged or refactor it to call the bulk implementation with one key.
- After successful bulk delete:
  - clear selection;
  - close the dialog;
  - refresh/derive counts through the existing vault state updates.

**RBAC**

- UI gate: bulk controls must depend on `canManageSecrets`.
- Backend gate: existing vault write checks remain the final authority. If a non-admin somehow triggers the action, the API should reject the write; the UI should show the existing error pattern.

**Confirmation dialog copy**

- Title: `Excluir secrets selecionadas`
- Description example: `Excluir 3 secrets do ambiente Production? Esta ação atualizará o vault criptografado e não poderá ser desfeita.`
- Confirm label: `Excluir 3 secrets`
- `destructive` enabled.

## Rollout Sequence

2. Add secret row checkbox props and tests without changing bulk behavior yet.
3. Add table/panel bulk toolbar and selection callbacks behind `canManageSecrets`.
4. Add bulk delete action in `useProjectSecrets` and wire confirmation dialog in `SecretsClient`.
5. Run focused tests, then full web unit/lint/build checks.
6. Perform mobile/desktop smoke test; capture screenshots if there is a visible UI change in the runnable app.
7. Update changelog/current task with final implementation notes after coding.

## Risks and Mitigations

- **Risk:** accidental secret deletion across environments.  
  **Mitigation:** selection is scoped to active environment and reset on environment change.
- **Risk:** multiple vault writes for bulk delete causing version conflicts.  
  **Mitigation:** filter selected keys and call `pushSecrets()` once.
- **Risk:** developers/viewers see destructive controls due to prop defaults.  
  **Mitigation:** default selection props to disabled/no-op and render bulk controls only when `canManageSecrets` is true.
- **Risk:** mobile notification panel still clipped by an ancestor with overflow.  
  **Mitigation:** use `position: fixed` on mobile so it is viewport-relative instead of ancestor-relative.
- **Risk:** checkbox styling diverges from app theme.  
  **Mitigation:** reuse existing Radix checkbox component if available; otherwise follow the current minimal black/gray controls used by role picker and dialogs.
